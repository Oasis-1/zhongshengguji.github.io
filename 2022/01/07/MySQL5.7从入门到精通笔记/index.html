JOIN

from 表1 inner join 表2 on表1.字段1=表2.字段1

  • JOIN: 如果表中有至少一个匹配，则返回行
  • LEFT JOIN: 即使右表中没有匹配，也从左表返回所有的行
  • RIGHT JOIN: 即使左表中没有匹配，也从右表返回所有的行
  • FULL JOIN: 只要其中一个表中存在匹配，就返回行

inner join内连接

注释：INNER JOIN 与 JOIN 是相同的。

INNER JOIN 关键字在表中存在至少一个匹配时返回行。如果 “Persons” 中的行在 “Orders” 中没有匹配，就不会列出这些行。

返回的结果集是通过on中共有字段可以组合成功的记录

共有的四个记录

left join 左连接

注释：在某些数据库中， LEFT JOIN 称为 LEFT OUTER JOIN。

LEFT JOIN 关键字会从左表 (Persons) 那里返回所有的行，即使在右表 (Orders) 中没有匹配的行。

返回的结果集是通过on中共有字段可以组合成功的记录，加上前面表的所有记录。匹配不成功的字段，会为null

结果为五条记录，有null值

right join 左连接

关于u的俩字段都为null

full join

返回的结果集是通过on中共有字段可以组合成功的记录，加上前面表的所有记录，再加上后面那个表的所有记录。匹配不成功的字段，会为null

UNION

UNION 操作符用于合并两个或多个 SELECT 语句的结果集。

请注意，UNION 内部的 SELECT 语句必须拥有相同数量的列。列也必须拥有相似的数据类型。同时，每条 SELECT 语句中的列的顺序必须相同。

union会自动去重

union all可以包含重复共有的记录

union前后两个结果集，然后这俩结果集字段结构需要相同，然后会返回一个合并过的结果集，至于要不要重复的记录，在于union和union all的差距

外键，不用

【强制】不得使用外键与级联，一切外键概念必须在应用层解决。 说明：（概念解释）学生表中的 student_id 是主键，那么成绩表中的 student_id 则为外键。如果更新学 生表中的 student_id，同时触发成绩表中的 student_id 更新，即为级联更新。外键与级联更新适用于单机 低并发，不适合分布式、高并发集群；级联更新是强阻塞，存在数据库更新风暴的风险；外键影响数据库 的插入速度。

外键的缺点主要是耦合度太高，改一个地方还要考虑其他关联的地方。

强阻塞应该是，外键更新其他表的时候，其他sql都得在后面等着，分布式传递消息本来需要网络，涉及到锁的知识点了，分布式

只有一个场景不能使用外键，就是分库分表，其它描述都不是真正原因。

性能问题？ 数据库的瓶颈在IO，不使用外键代码里做数据完整性检查，磁盘IO省不了，网络IO占用增加。

高并发锁的问题？一般数据库都不会启用读锁，例如MYSQL事务级别设置为读提交，个人认为没有理由使用其它事务级别。

扩展性问题？ 主流数据库均支持外键约束。

数据库水平扩展不易？ 64核cpu已发布，最大支持4T内存，再次强调数据库性能瓶颈在磁盘，无论是否使用外键，只要要做数据完整性检查，磁盘IO是省不了的。读写分离也是很成熟的方案。

外键总结

https://juejin.cn/post/6956895323922890783/

  1. 阿里提出：强制不能使用外键，外键概念必须在应用层解决
  2. 外键的优点：保证关联的表的事务的一致性，增删改操作都会收到约束，保证一直
  3. 缺点：分布式锁中的问题，还有分库分表（数据源不同）

查看支持的引擎

数据库引擎

InnoDB

事务型数据库的首选引擎，支持事务安全表（ACID）行锁定，外键

5.5.5之后InnoDB就是默认，主要特征有：

  1. 支持事务
  2. InnoDB是为处理巨大数据量的最大性能设计。它的CPU效率可能是任何其他基于磁盘的关系数据库引擎所不能匹敌的。
  3. InnoDB存储引擎完全与Mysql服务器整合，innodb在主内存中缓存数据和索引，而维持自己的缓冲池。
  4. 支持外键

MyISAM

性能极佳，不支持事务

MyISAM是基于ISAM的存储引擎。

web，数据存储最常用的之一

插入，查询速度块。不支持事务

5.5.5之前是mysql的默认引擎。

与innoDB的区别

https://blog.csdn.net/qq_35642036/article/details/82820178

  1. 事务

  2. myisam不支持外键

  3. 索引，innoDB是B+Tree作为索引的数据文件和索引在一起的

  4. 行数，count时，innodb是全表扫描myisam有个变量保存了行数，但是不能加where

  5. Innodb不支持全文索引，而MyISAM支持全文索引，在涉及全文索引领域的查询效率上MyISAM速度更快高；PS：5.7以后的InnoDB支持全文索引了

  6. MyISAM表格可以被压缩后进行查询操作

  7. InnoDB支持表、行(默认)级锁，而MyISAM支持表级锁

  8. InnoDB表必须有唯一索引（如主键）（用户没有指定的话会自己找/生产一个隐藏列Row_id来充当默认主键），而Myisam可以没有

  9. Innodb存储文件有frm、ibd，而Myisam是frm、MYD、MYI

    ​ Innodb：frm是表定义文件，ibd是数据文件

    ​ Myisam：frm是表定义文件，myd是数据文件，myi是索引文件

MEMORY

数据放在内存中的引擎

MEMORY默认使用哈希索引。速度比使用B型树索引快。当然如果你想用B型树索引，可以在创建索引时指定。

注意，MEMORY用到的很少，因为它是把数据存到内存中，如果内存出现异常就会影响数据。如果重启或者关机，所有数据都会消失。因此，基于MEMORY的表的生命周期很短，一般是一次性的。

https://www.cnblogs.com/yuxiuyan/p/6511837.html
